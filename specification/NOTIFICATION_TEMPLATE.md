# 通知模板

## 目标

本模板用于统一社区项目中的通知写法，先保证：

- 覆盖常见场景
- 结构基本一致
- 内容简单明了
- 能直接复制修改后发送

这份文档优先解决“先有一个基本可用的通知模板库”，后续可以再按渠道、系统、严重级别继续细化。

## 使用原则

- 先写结论，再写细节
- 优先写清影响范围、当前状态、下一步动作
- 不写模糊表述，例如“可能有点异常”“稍后恢复”
- 能量化的内容尽量量化，例如时间、范围、模块、版本
- 面向外部用户和面向内部成员的语气要分开

## 通知最小结构

大多数通知至少应包含以下信息：

- 通知标题
- 事件类型
- 发生时间
- 影响范围
- 当前状态
- 处理动作或下一步安排
- 责任人或跟进人

## 发送前检查

- 标题是否一眼能看懂
- 是否写清楚“谁会受影响”
- 是否写清楚“现在是什么状态”
- 是否写清楚“接下来要做什么”
- 是否写清楚“什么时候再同步”

## 一、发布通知模板

适用场景：

- 新版本发布
- 模块上线
- 配置变更上线
- 功能灰度发布

### 模板 1：标准发布通知

```text
【发布通知】<项目名>/<模块名> 已发布

发布时间：YYYY-MM-DD HH:mm
发布版本：vX.Y.Z
发布范围：<生产 / 测试 / 指定环境>

本次变更：
1. <变更点 1>
2. <变更点 2>
3. <变更点 3>

影响说明：
- 是否影响现有用户：
- 是否需要重新登录/刷新/重启：
- 是否涉及配置更新：

验证状态：
- 已完成基础验证
- 已确认核心功能可用

跟进人：@xxx
```

### 模板 2：简版发布通知

```text
【构建完成】<项目名>/<模块名> vX.Y.Z

时间：YYYY-MM-DD HH:mm
环境：编译节点
内容：<一句话说明本次发布重点>
状态：已构建，上传完毕
跟进人：
```

### 模板 3：灰度发布通知

```text
【灰度发布通知】<项目名>/<模块名>

时间：YYYY-MM-DD HH:mm
版本：vX.Y.Z
灰度范围：<用户范围 / 节点范围 / 流量比例>

本次目标：
- <目标 1>
- <目标 2>

观察指标：
- <指标 1>
- <指标 2>

当前状态：
- 已开始灰度
- 暂无异常 / 发现异常已处理中

下一次同步时间：YYYY-MM-DD HH:mm
跟进人：@xxx
```

## 二、系统错误通知模板

适用场景：

- 服务异常
- 接口错误率升高
- 节点故障
- 任务执行失败
- 依赖服务不可用

### 模板 1：内部错误通知

```text
【异常通知】<项目名>/<模块名>

发现时间：YYYY-MM-DD HH:mm
异常等级：P1 / P2 / P3
当前状态：处理中 / 已定位 / 已恢复 / 持续观察

异常现象：
- <现象 1>
- <现象 2>

影响范围：
- 影响用户：
- 影响功能：
- 影响环境：

当前判断：
- <初步原因或现阶段判断>

已执行操作：
1. <操作 1>
2. <操作 2>

下一步动作：
1. <下一步 1>

下一次同步时间：YYYY-MM-DD HH:mm
负责人：
```

### 模板 2：对外错误通知

```text
【服务异常通知】<项目名>

时间：YYYY-MM-DD HH:mm

我们发现部分功能出现异常，当前可能影响：
- <功能 1>
- <功能 2>

当前状态：
- 已开始处理
- 正在持续排查和恢复

建议：
- <是否建议稍后重试>
- <是否建议暂时避免某些操作>

后续进展将持续同步。
```

### 模板 3：恢复通知

```text
【恢复通知】<项目名>/<模块名>

异常时间：YYYY-MM-DD HH:mm
恢复时间：YYYY-MM-DD HH:mm
当前状态：已恢复

影响范围：
- <影响描述>

原因摘要：
- <一句话说明原因>

处理结果：
- <一句话说明处理动作>

后续动作：
- 持续观察
- 补充复盘 / 修复 / 防护

负责人：@xxx
```

## 五、推荐标题格式

建议统一使用：

- `【发布通知】项目名/模块名`
- `【系统异常】项目名/模块名`
- `【恢复通知】项目名/模块名`
- `【重要变更通知】项目名/事项名`
- `【维护通知】项目名/模块名`
- `【运维操作通知】项目名/模块名`

这样做的好处是：

- 一眼能看出通知类型
- 便于群消息检索
- 便于后续脚本自动分类

## 六、AI 编写建议

考虑到现在很多通知会先让大模型起草，建议这样使用：

- 先提供事实，再让模型整理
- 不要让模型猜测影响范围
- 不要让模型编造恢复时间、原因和负责人
- 对外通知要比内部通知更克制，避免过度技术细节

推荐给模型的输入信息至少包含：

- 事件类型
- 发生时间
- 影响范围
- 当前状态
- 已执行动作
- 下一步动作
- 负责人

## 七、提示词模板

### 提示词 1：生成发布通知

```text
请基于以下事实，生成一份简单明了的发布通知：
- 项目名：
- 模块名：
- 发布时间：
- 发布版本：
- 发布环境：
- 变更点：
- 影响范围：
- 验证状态：
- 跟进人：

要求：
- 使用正式、简洁的中文
- 先写结论，再写细节
- 不要编造不存在的影响和验证结果
```

### 提示词 2：生成系统错误通知

```text
请基于以下事实，生成一份系统异常通知：
- 项目名：
- 模块名：
- 发现时间：
- 异常等级：
- 当前状态：
- 异常现象：
- 影响范围：
- 已执行操作：
- 下一步动作：
- 下一次同步时间：
- 负责人：

要求：
- 明确写出影响范围和当前状态
- 不要虚构根因
```

```text
- 结构清晰
```
