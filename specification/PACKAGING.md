# 打包规范

## 目标

统一项目的打包、发布和启动验证流程，确保开发者提交的安装包在干净环境下也能够正常启动，避免仅在本地开发目录可用、换机器或重新克隆后失效的问题。

## 基本要求

- 如果打包脚本和启动脚本在本地测试没有问题，仍然必须再找一台机器，或至少在一个全新的目录重新克隆代码后重新验证。
- 验证时必须假设目标环境没有继承当前开发目录中的临时文件、本地缓存和手工修补内容。
- 打包流程应尽可能自动化，减少手工操作和环境依赖。
- 每次打包前必须先同步远端最新代码，确保当前打包基于最新主线，而不是本地落后的代码状态。
- 如果项目包含前端 `web` 目录，打包前必须重新安装前端依赖，确保使用的是最新依赖状态。

## 启动脚本规范

项目必须提供启动脚本 `scripts/starter.sh`，并满足以下要求：

- 脚本支持 `start`、`stop`、`restart` 参数。
- 无参数执行时，默认行为等同于 `start`。
- 脚本应能够在安装包解压后的目录中直接运行，不依赖开发目录的额外文件。
- 脚本应优先使用相对路径定位当前项目目录，避免写死开发机绝对路径。

建议示例：

```bash
./scripts/starter.sh
./scripts/starter.sh start
./scripts/starter.sh stop
./scripts/starter.sh restart
```

## 打包脚本规范

项目必须提供打包脚本 `scripts/package.sh`，并满足以下要求。

### 输出目录

- 打包产物统一输出到项目根目录下的 `output` 目录。
- 项目的 `.gitignore` 中必须添加 `output/`，避免打包产物进入版本控制。

### 打包前同步要求

执行 `scripts/package.sh` 前，脚本应主动同步远端最新代码。最低要求如下：

- 获取远端最新提交信息，例如执行 `git fetch --tags origin`。
- 同步 `main` 分支最新代码后再执行版本比较和打包逻辑。
- 如果当前工作区有未提交改动，脚本应直接终止，并提示开发者先清理工作区。
- 如果脚本在同步或打包过程中切换到了其他分支或 TAG，结束后必须切回执行前所在分支。

建议流程：

```bash
git fetch --tags origin
git checkout main
git pull --rebase origin main
```

### 安装包内容

安装包中应至少包含以下内容：

- 最新的二进制文件
- 配置模板
- 前端静态资源
- 启动脚本 `scripts/starter.sh`
- 启动所需的其他必要资源文件

要求是：解压安装包后，进入目录即可直接执行 `scripts/starter.sh` 完成启动验证。

### 前端依赖处理

如果项目包含前端 `web` 目录，打包脚本必须在打包前重新安装依赖，再构建或收集静态资源。

最低要求如下：

- 进入 `web` 目录执行依赖安装。
- 确保依赖安装基于当前最新代码状态。
- 依赖安装失败时，打包脚本应直接退出，不能继续生成安装包。

建议命令：

```bash
cd web
npm install
```

### 安装包命名规则

安装包文件名格式：

```text
<project-name>-<tag-name>-<short-hash>.tar.gz
```

示例：

```text
webdav-v1.0.1-8a54401.tar.gz
```

命名规则说明：

- `<project-name>`：项目名称，通常取当前仓库目录名。
- `<tag-name>`：本次打包对应的 Git Tag。
- `<short-hash>`：最近一次提交的 7 位短哈希值。

## TAG 规则

`scripts/package.sh` 必须支持可选的 `TAG` 参数。

TAG 命名格式：

```text
v<major>.<minor>.<patch>
```

示例：

```text
v1.0.1
```

## 打包逻辑

### 传入 TAG 参数

执行方式示例：

```bash
./scripts/package.sh v1.0.1
```

要求如下：

- 如果指定的 TAG 不存在，则不打包，并给出明确提示。
- 如果指定的 TAG 存在，则切换到该 TAG 对应的代码状态进行打包。
- 打包完成后，如果脚本在过程中切换了分支或代码状态，必须切回执行前所在的分支。

### 不传 TAG 参数

执行方式示例：

```bash
./scripts/package.sh
```

要求如下：

1. 获取当前项目中最大的 TAG。
2. 比较该最大 TAG 对应提交与当前 `main` 分支最近一次提交的哈希值是否一致。
3. 如果一致，则不打包，并给出明确提示，因为当前 `main` 已经对应一个已发布版本。
4. 如果不一致，则在最大 TAG 的 `patch` 版本号上加 `1`。
5. 基于新的版本号创建新 TAG。
6. 将新 TAG 推送到远端。
7. 切换到新的 TAG 进行打包。
8. 打包完成后切回原始分支。

例如：

- 当前最大 TAG 为 `v1.0.1`
- `main` 最新提交未发布
- 则应自动创建 `v1.0.2`
- 推送新 TAG 后再基于 `v1.0.2` 打包

## 脚本行为要求

`scripts/package.sh` 应尽可能自动处理各种依赖问题，包括但不限于：

- 自动同步远端最新代码和最新 TAG
- 自动创建缺失目录，例如 `output/`
- 自动检查打包工具是否可用，例如 `tar`
- 自动检查 Git 工作区和当前分支状态
- 如果存在 `web` 目录，自动执行 `npm install`
- 自动处理分支切换和恢复
- 在必要时输出明确的错误信息和下一步建议

如果脚本需要切换分支、切换到某个 TAG，或修改当前 Git 状态，必须保证在脚本结束时恢复到执行前的分支或引用状态。

## 开发者验收要求

开发者完成打包后，必须验证安装包是否能够在干净环境下正常启动。

最低验收要求如下：

1. 重新克隆项目，或在新的空目录中获取完整代码。
2. 执行打包脚本，生成安装包。
3. 解压安装包。
4. 进入解压后的目录。
5. 执行 `scripts/starter.sh`。
6. 确认服务能够正常启动。

建议验证命令：

```bash
tar -xzf output/<package-name>.tar.gz
cd <package-name>
./scripts/starter.sh
```

## 禁止事项

- 不要只在当前开发目录验证打包结果。
- 不要依赖未提交的本地文件参与打包。
- 不要将 `output/` 打包结果提交到 Git 仓库。
- 不要在脚本切换分支后不恢复现场。
- 不要生成无法独立启动的安装包。
